O que é Flutter e Dart?

Dart é a linguagem de programação. Pense nela como um JavaScript tipado — tem classes, async/await, e roda tanto no celular quanto na web.

Flutter é o framework que usa Dart pra construir telas. Tudo no Flutter é um Widget — um botão é um widget, um texto é um widget, a tela inteira é um widget. Widgets se encaixam como peças de LEGO pra montar a interface.

dart 
// Exemplo: um texto na tela
Text('Olá, mundo!')

// Exemplo: um botão
ElevatedButton(
  onPressed: () => print('clicou!'),
  child: Text('Clique aqui'),
)

A Arquitetura: Clean + BLoC

O projeto usa dois conceitos juntos:

Clean Architecture
Separação de responsabilidades

Como organizamos as pastas e garantimos que cada parte do código faça apenas o que é sua responsabilidade.

BLoC Pattern
Gerenciamento de estado

Como gerenciamos o estado da tela — loading, erro, sucesso, dados carregados, etc.

A Analogia do Restaurante

Imagine que o app é um restaurante. Cada feature (login, cadastro, dashboard) é um prato do cardápio. Pra preparar um prato, temos 3 setores:

presentation/

Garçom anota o pedido e traz o prato. O BLoC é o garçom: recebe pedidos e entrega resultados.

Telas Botões Formulários BLoC (garçom)
domain/
📋 Cardápio + Regras

O que o restaurante sabe fazer. "Temos estrogonofe? Sim." Sem saber como é preparado.

Entities UseCases Repository (contrato)
data/
🍳 Cozinha — Trabalho pesado

Pega ingredientes e cozinha. De onde vêm os dados e como processá-los de fato.

DataSources (API, cache) Models (JSON) Repository (impl)
✦ Regra de ouro: o salão nunca entra na cozinha. O cardápio não sabe como o prato é feito. Cada setor só conhece o setor diretamente abaixo dele.

Camadas na Prática

Vamos acompanhar o que acontece quando o usuário faz login com email e senha, passo a passo.

1
Usuário clica "Entrar" → Tela dispara evento

A tela (login_page.dart) captura a ação e envia um evento pro BLoC:

dart — login_page.dart
// Na tela, quando o usuário clica em "Entrar":
context.read<LoginBloc>().add(
  LoginSubmitted(
    email: 'joao@email.com',
    password: '123456',
  ),
);
💬

"Ei garçom, o cliente quer fazer login com esse email e senha."

2
BLoC recebe o evento e chama o UseCase

O BLoC é o garçom. Ele recebe o pedido, emite LoginLoading e chama o UseCase:

dart — login_bloc.dart
@injectable
class LoginBloc extends Bloc<LoginEvent, LoginState> {
  LoginBloc(this._loginWithEmail)
      : super(const LoginInitial()) {
    on<LoginSubmitted>(_onLoginSubmitted);
  }
  final LoginWithEmail _loginWithEmail;

  Future<void> _onLoginSubmitted(
    LoginSubmitted event,
    Emitter<LoginState> emit,
  ) async {
    emit(const LoginLoading());  // mostra o loading
    try {
      final user = await _loginWithEmail(
        email: event.email,
        password: event.password,
      );
      emit(LoginSuccess(user));  // deu certo!
    } on ApiException catch (e) {
      emit(LoginFailure(e.message));  // deu erro
    }
  }
}
3
UseCase chama o Repository

O UseCase representa a ação de negócio. Agora é simples, mas é onde lógica complexa viria no futuro:

dart — login_with_email.dart
@lazySingleton
class LoginWithEmail {
  const LoginWithEmail(this._repository);
  final AuthRepository _repository;

  Future<User> call({
    required String email,
    required String password,
  }) => _repository.loginWithEmail(email: email, password: password);
}
💡

Mesmo sendo simples agora, num futuro o UseCase pode orquestrar: autenticar + salvar token + buscar perfil + registrar dispositivo para push — tudo sem poluir o BLoC.

4
Repository abstrato define o contrato

O contrato diz o que a cozinha sabe fazer, sem dizer como:

dart — auth_repository.dart
abstract class AuthRepository {
  Future<User> loginWithEmail({
    required String email,
    required String password,
  });
  Future<void> logout();
  Future<User?> getCurrentUser();
}
5
RepositoryImpl faz o trabalho real
dart — auth_repository_impl.dart
@LazySingleton(as: AuthRepository)
class AuthRepositoryImpl implements AuthRepository {
  const AuthRepositoryImpl(this._remoteDataSource, this._localDataSource);
  final AuthRemoteDataSource _remoteDataSource;
  final AuthLocalDataSource _localDataSource;

  @override
  Future<User> loginWithEmail({
    required String email,
    required String password,
  }) async {
    final result = await _remoteDataSource.login(email: email, password: password);
    await _localDataSource.saveToken(result.token); // mantém o usuário logado
    return result.user;
  }
}
6
DataSource faz a chamada HTTP
dart — auth_remote_datasource.dart
@lazySingleton
class AuthRemoteDataSource {
  const AuthRemoteDataSource(this._apiClient);
  final ApiClient _apiClient;

  Future<LoginResponseModel> login({
    required String email,
    required String password,
  }) async {
    final response = await _apiClient.dio.post(
      ApiEndpoints.login,
      data: {'email': email, 'password': password},
    );
    return LoginResponseModel.fromJson(response.data);
  }
}
7
Model traduz o JSON, Entity é o objeto puro

O JSON da API chega assim:

json — resposta da API
{
  "token": "eyJhbGciOiJIUzI1NiIs...",
  "user": {
    "id": "abc123",
    "name": "João Silva",
    "email": "joao@email.com"
  }
}

O Model sabe ler esse JSON. A Entity é o objeto puro que sobe limpo pro BLoC:

dart — user.dart (entity) vs user_model.dart
// domain/entities/user.dart — OBJETO PURO, sem JSON
class User extends Equatable {
  const User({required this.id, required this.name, required this.email});
  final String id, name, email;
  @override
  List<Object?> get props => [id, name, email];
}

// data/models/user_model.dart — EXTENDS User, sabe ler JSON
class UserModel extends User {
  const UserModel({required super.id, required super.name, required super.email});

  factory UserModel.fromJson(Map<String, dynamic> json) {
    return UserModel(
      id: json['id'] as String,
      name: json['name'] as String,
      email: json['email'] as String,
    );
  }
}
🤔

Por que dois arquivos? A Entity pertence ao domain/ (não depende de nada) e o Model pertence ao data/ (sabe ler JSON). O Model extends a Entity — ele É um User, mas com superpoderes de parsing.

BLoC: Eventos & Estados

O BLoC funciona como um walkie-talkie entre a tela e os dados.

Eventos — o que o usuário faz

dart — login_event.dart
// Classe base selada — só essas subclasses existem
sealed class LoginEvent extends Equatable {
  const LoginEvent();
  @override
  List<Object?> get props => [];
}

// "Quero fazer login"
class LoginSubmitted extends LoginEvent {
  const LoginSubmitted({required this.email, required this.password});
  final String email, password;
  @override
  List<Object?> get props => [email, password];
}

Estados — o que a tela mostra

LoginInitial

Tela parada, formulário vazio. Estado inicial antes de qualquer interação.

LoginLoading

Mostra spinner, desabilita o botão. Aguardando resposta da API.

LoginSuccess(user)

Deu certo! Contém o objeto User autenticado.

LoginFailure(message)

Deu erro. Contém a mensagem de erro pra exibir na tela.

🖼️

Como a Tela Usa o BLoC

dart — login_page.dart
class LoginPage extends StatelessWidget {
  @override
  Widget build(BuildContext context) {
    return BlocProvider(
      // Cria o BLoC via injeção de dependência
      create: (_) => getIt<LoginBloc>(),
      child: const _LoginView(),
    );
  }
}

// BlocListener → "side-effects" (navegação, snackbar)
BlocListener<LoginBloc, LoginState>(
  listener: (context, state) {
    if (state is LoginSuccess) {
      context.go('/dashboard');         // navega pro dashboard
    } else if (state is LoginFailure) {
      ScaffoldMessenger.of(context)
          .showSnackBar(SnackBar(content: Text(state.message)));
    }
  },
  child: ...
)

// BlocBuilder → reconstrói o botão baseado no estado
BlocBuilder<LoginBloc, LoginState>(
  builder: (context, state) {
    return ElevatedButton(
      onPressed: state is LoginLoading ? null : () {
        context.read<LoginBloc>().add(LoginSubmitted(
          email: _emailController.text,
          password: _passwordController.text,
        ));
      },
      child: state is LoginLoading
          ? const CircularProgressIndicator()
          : const Text('Entrar'),
    );
  },
)

Injeção de Dependência

O BLoC recebe o UseCase, que recebe o Repository, que recebe o DataSource... mas ninguém faz new manualmente. Usamos get_it + injectable.

Anotações disponíveis

dart — anotações injectable
@injectable          // cria instância NOVA toda vez (usado em BLoCs)
@lazySingleton       // cria UMA vez e reutiliza (usado em UseCases, DataSources)
@LazySingleton(as: AuthRepository)  // registra pela interface

A cadeia automática

getIt<LoginBloc>()
└─ precisa de LoginWithEmail
└─ precisa de AuthRepository
└─ resolve para AuthRepositoryImpl
└─ precisa de AuthRemoteDataSource + AuthLocalDataSource
└─ RemoteDS precisa de ApiClient (Dio)
└─ LocalDS precisa de SharedPreferences
✨

Tudo isso é resolvido automaticamente. Na tela, basta getIt<LoginBloc>() — o framework monta toda a cadeia.

Gerar o código de DI

$ dart run build_runner build --delete-conflicting-outputs

Isso atualiza o arquivo lib/core/di/injection.config.dart com todas as dependências mapeadas.

Tratamento de Erros

A API pode falhar de várias formas. Temos uma hierarquia de exceções em core/network/api_exception.dart:

ApiException (base selada)
UnauthorizedException ← 401 — email/senha errados
NotFoundException ← 404
RateLimitedException ← 429 — muitas tentativas
TimeoutException ← timeout de conexão
NetworkException ← sem internet
ServerException ← 500+
UnknownApiException ← qualquer outro erro

O ApiClient intercepta erros do Dio e converte pra essas exceções. O BLoC captura com on ApiException catch (e) e emite o estado de falha com a mensagem correta.

Criando uma Feature Nova

Quer criar a feature "Cadastro de Usuário"? Siga esse checklist:

1. Crie a estrutura de pastas

lib/features/register/
├── data/
│   ├── datasources/
│   ├── models/
│   └── repositories/
├── domain/
│   ├── entities/
│   ├── repositories/
│   └── usecases/
└── presentation/
    ├── bloc/
    ├── pages/
    └── widgets/

2. Comece pelo domain/ (de dentro pra fora)

dart — domain/
// domain/entities/register_result.dart
class RegisterResult extends Equatable {
  final String userId, email;
  // campos puros, sem JSON
}

// domain/repositories/register_repository.dart
abstract class RegisterRepository {
  Future<RegisterResult> register({
    required String name, required String email, required String password,
  });
}

// domain/usecases/register_user.dart
@lazySingleton
class RegisterUser {
  const RegisterUser(this._repository);
  final RegisterRepository _repository;

  Future<RegisterResult> call({...}) => _repository.register(...);
}

3. Implemente o data/

dart — data/
// data/models/register_result_model.dart
class RegisterResultModel extends RegisterResult {
  factory RegisterResultModel.fromJson(Map<String, dynamic> json) { ... }
}

// data/datasources/register_remote_datasource.dart
@lazySingleton
class RegisterRemoteDataSource {
  // chamada HTTP real: POST /auth/register
}

// data/repositories/register_repository_impl.dart
@LazySingleton(as: RegisterRepository)
class RegisterRepositoryImpl implements RegisterRepository {
  // delega pro datasource
}

4. Crie o BLoC e rode o build_runner

✓

✓
Criar register_state.dart com Initial, Loading, Success, Failure
✓
Criar register_bloc.dart seguindo o padrão do LoginBloc
✓
Rodar dart run build_runner build --delete-conflicting-outputs
✓
Adicionar a rota em core/router/app_router.dart

TL;DR — Resumão

01
Três pastas por feature

data/ (cozinha), domain/ (cardápio), presentation/ (salão)

02
BLoC no presentation/

Recebe eventos da tela, chama UseCases, emite estados de volta

03
UseCase no domain/

Representa uma ação de negócio. Onde lógica complexa fica centralizada

04
Repository: contrato + impl

Contrato no domain/ (o quê), implementação no data/ (como)

05
DataSource no data/

Faz a chamada HTTP real ou acessa armazenamento local

06
Model vs Entity

Model traduz JSON, Entity é o objeto puro. Model extends Entity

07
Injeção automática

get_it + injectable montam toda a cadeia de dependências automaticamente

08
Feature nova = copy-paste

Copie a estrutura de uma feature existente e siga o padrão estabelecido
